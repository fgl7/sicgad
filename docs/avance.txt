1. AVANCE SICGAD PARA DATA DIARIO Y CERTIFICACION MENSUAL

A. Mejoras en la interfaz y experiencia de uso

- Se reorganizó la presentación visual del sistema, restringiendo el inicio de sesión solo a personal registrado. Temporalmente se deshabilitó el acceso sin registro a la visualización de datos publicados mientras se completan las pruebas, lo que ayuda a proteger la información sensible. (colocar captura de la pantalla de inicio de sesión con acceso restringido)
- Se implementó una barra lateral de navegación (sidebar) con secciones que se muestran u ocultan según el tipo de usuario. Por ejemplo, el administrador ve todas las secciones (usuarios, esquemas, carga, validación, indicadores y auditoría), mientras que el cargador solo ve las opciones relacionadas con la carga y revisión de sus datos. Esto reduce la complejidad y evita que los usuarios se confundan con opciones que no necesitan. (colocar captura del sidebar visto por un administrador y por un cargador)
- Se mejoró la interfaz para cada tipo de usuario (cargadores, validadores y administradores), adaptando formularios, menús y ventanas modales para que el flujo sea más claro y ordenado. Se trabajó especialmente en los mensajes y botones para que cada usuario sepa en todo momento qué paso sigue (por ejemplo, “revisar borrador”, “enviar a validación” o “aprobar certificación”). (colocar captura de un formulario de carga y de una ventana de aprobación)
- Se añadieron tablas de visualización de datos en estado “borrador”, de manera que los validadores puedan revisar la información preliminar antes de aprobarla o rechazarla, sin necesidad de que esté publicada. Esto permite discutir y corregir datos internamente antes de que aparezcan como información oficial. (colocar captura de la tabla con datos en estado borrador)
- Se ajustó la gestión de usuarios para permitir definir perfiles a nivel global (todas las plantas) o por planta específica, y además indicar si cada validador puede revisar datos diarios, datos mensuales o ambos. Con esto se refleja de mejor manera la realidad organizacional de YLB y MHE, donde hay roles con alcance nacional y otros centrados en una planta concreta.
- Se habilitó la visualización de datos en borrador para los usuarios que participan en la carga y validación (cargadores y validadores), manteniendo el acceso a datos oficiales solo para quienes corresponda. Así, los equipos pueden detectar inconsistencias a tiempo sin exponer información incompleta a terceros.
- Se reforzó el flujo de aprobación según institución, diferenciando claramente los roles de YLB y del Ministerio de Hidrocarburos y Energías (MHE). En particular, se configuró el sistema para que la certificación mensual solo se considere “publicada” cuando se cumplan los pasos de aprobación definidos para ambas instituciones. (colocar captura de la pantalla de detalle de certificación con los niveles de aprobación YLB/MHE)
- Se implementó un sistema de alertas de validación que hoy está en etapa de prueba con varios validadores. Estas alertas aparecen como indicadores visibles en la interfaz (por ejemplo, en la barra lateral y en la bandeja de validación) y ayudan a que cada usuario vea cuántos conjuntos de datos tiene pendientes. (colocar captura de la bandeja de validación mostrando las alertas)
- Se completó la funcionalidad para crear esquemas de certificación mensual de producción, junto con su flujo de carga y validación de datos. Además, cuando se completa la carga y aprobación de todos los días de un mes, el sistema genera automáticamente el esquema mensual de certificación correspondiente, basado en los datos diarios ya aprobados. Esto reduce el trabajo manual y mejora la trazabilidad entre los datos diarios y la certificación mensual. (colocar captura del esquema de certificación mensual generado a partir de datos diarios)

Pruebas funcionales con datos reales

- Carga de datos diarios:  
  Se realizaron pruebas del flujo completo para la carga diaria, incluyendo creación y validación del esquema, carga de archivos, detección de errores, aprobación/rechazo, publicación, visualización y auditoría de acciones. Estas pruebas permitieron identificar errores en el código, sobre los que ya se trabajó para aplicar correcciones y ajustes, buscando que el flujo sea estable desde la carga hasta la publicación. (colocar captura de la pantalla de carga diaria y del resumen de errores)  
  La información que antes se almacenaba únicamente en archivos Excel dentro del sistema ahora se guarda en una base de datos interna (SQLite3) durante la etapa de desarrollo. Este cambio facilita el análisis histórico, la búsqueda de problemas y la futura integración con tableros de indicadores (KPIs). Está previsto migrar esta base de datos a PostgreSQL cuando el sistema pase a un entorno de producción. (colocar captura de un ejemplo de datos diarios ya publicados en el tablero de indicadores)

- Carga de datos mensuales y certificación mensual de producción:  
  Se están ejecutando pruebas del flujo completo para la certificación mensual, utilizando datos de ejemplo y escenarios similares a los que se presentarán en la operación real. Este flujo incluye:
  - La consolidación automática de los datos diarios en un resumen mensual.
  - La revisión por parte de distintos niveles de validadores (por ejemplo, responsables de planta, niveles superiores en YLB y representantes de MHE).
  - La publicación final de la certificación mensual, que será la referencia oficial para reportes y cálculo de regalías.  
  Durante estas pruebas se siguen identificando detalles de comportamiento (por ejemplo, qué ocurre cuando falta la aprobación de un nivel o cuando hay cambios de último momento en los datos diarios), que se están corrigiendo para garantizar la estabilidad del flujo completo. Actualmente, los datos se guardan en SQLite3, con la previsión de migrar a PostgreSQL en la etapa de producción. A la fecha de este informe, las pruebas continúan en coordinación con los equipos funcionales. (colocar captura de la pantalla de revisión de certificación mensual con el resumen consolidado)

Mejora de validaciones y reglas de aprobación

- Se optimizaron los mensajes de error para que sean más claros para los usuarios no técnicos, reemplazando textos genéricos por explicaciones que indican qué campo tiene el problema y cómo puede corregirse. (colocar captura de un ejemplo de mensaje de error en la pantalla de carga)
- Se ajustaron criterios de verificación y pasos del flujo de aprobación, con el objetivo de reforzar la calidad y precisión de los datos reportados. Por ejemplo, se revisa que no existan periodos duplicados, que los estados de los datos sigan el orden correcto y que no se pueda publicar información con errores pendientes. (colocar captura del flujo de estados de un dataset desde borrador hasta publicado)
- Se alineó el comportamiento del sistema con el modelo de validación por niveles, de manera que los distintos roles (por ejemplo, niveles de YLB y MHE) intervengan en el orden previsto. Esto evita que se salten pasos y garantiza que la certificación mensual cuente con el visto bueno de todas las instancias necesarias. (colocar captura del historial de aprobaciones mostrando los distintos niveles)
- Se reforzó el registro de acciones (auditoría) para que quede constancia de quién cargó los datos, quién los aprobó o rechazó y en qué momento. Este historial será clave para aclarar dudas futuras y para cumplir con los requisitos de trazabilidad y control. (colocar captura de la pantalla de auditoría con el listado de eventos)

2. AVANCE CUANTIFICACIÓN DEL RENDIMIENTO PRODUCTIVO Y LA EFICIENCIA DE PROCESOS

En el marco de la propuesta metodológica para medir rendimiento productivo y eficiencia de procesos (plantas PCS, KCl y Li2CO3), se inició la construcción del componente de “Desempeño” dentro de SICGAD. El objetivo de este avance es que el sistema no solo reciba y valide datos, sino que también pueda transformar esa información en indicadores mensuales comparables, con trazabilidad y posibilidad de actualización cuando cambien los esquemas de datos.

Principales mejoras implementadas hasta el momento:

- Se incorporó una nueva sección “Desempeño” en la interfaz (visible para el usuario Administrador del sistema) para centralizar el trabajo relacionado con rendimiento y eficiencia.
- Se definió que el usuario Administrador (rol ADMIN dentro del sistema, no el superusuario técnico) será el único responsable de realizar estas asignaciones de variables. Esto ayuda a mantener un criterio institucional único y reduce el riesgo de que existan cálculos distintos por cada usuario.
- Se implementó un enfoque flexible de “asignación de variables”, donde el Administrador puede vincular las variables de las fórmulas (por ejemplo Msales(m), Msalmuera(m-Δt), Xsolids(m-Δt)) con campos existentes en los esquemas diarios o mensuales de la planta. Esto permite adaptar el cálculo sin necesidad de modificar el código cuando se actualizan nombres o estructuras de los esquemas.
- Para esta asignación, el Administrador puede trabajar por planta (por ejemplo PCS) y ver una lista de campos disponibles en los esquemas ya definidos en el sistema, para seleccionar cuáles alimentan cada variable de la fórmula.
- Se preparó el sistema para manejar resultados en dos estados:
  - “Draft”: resultados preliminares a partir de datos cargados/publicados durante el mes (sin esperar el cierre mensual).
  - “Certificado”: resultados a partir de datos mensuales certificados y aprobados, usados como referencia oficial.
- Se desarrolló un primer MVP para la Planta de Concentración de Sales (PCS) enfocado en la Fórmula 1 (Rendimiento de Producción Mensual %). En esta pantalla el Administrador:
  - Selecciona el mes a evaluar.
  - Asigna las variables requeridas por la fórmula a campos reales del sistema (por planta).
  - Visualiza un resultado estimado (preview) para Draft y Certificado.
  - Cuenta con un gráfico inicial para ver la tendencia mensual.
- Se iniciaron pruebas funcionales (backend y frontend) para verificar el funcionamiento correcto de esta primera implementación (Fórmula 1), con el objetivo de usarla como ejemplo y referencia para el resto de fórmulas del documento.
- Se sentaron bases para el “catálogo” de variables e indicadores, de modo que el sistema pueda ir incorporando progresivamente las demás fórmulas del documento (recuperación, energía, balances, pérdidas, etc.) manteniendo una lógica uniforme.
- Se ajustó la gestión de dependencias para asegurar compatibilidad con carga y lectura de archivos Excel, ya que SICGAD trabaja con plantillas de carga y materialización de datos.

Este avance deja listo el camino para la siguiente etapa: alimentar la Fórmula 1 con datos reales (tanto diarios como mensuales certificados), ajustar el criterio de cálculo (por ejemplo sumas por día, selección de columnas correctas y desfase Δt) y extender el mismo mecanismo a las siguientes fórmulas de desempeño.

Lo que falta (siguientes pasos):

- Ajustar y validar el criterio final de la Fórmula 1, definiendo claramente para cada variable si se alimentará desde sumas/agrupaciones diarias (avance del mes) o desde valores del mensual certificado (fuente oficial).
- Completar la visualización de KPIs para que se vea claramente el indicador en estado Draft y el indicador en estado Certificado cuando ya exista el mensual aprobado (incluyendo comparación de ambos valores y su tendencia).
- Asegurar que el indicador “Certificado” se actualice automáticamente cuando se publique o bloquee la certificación mensual, para que el resultado refleje siempre la versión oficial.
- Mejorar la trazabilidad visible para el Administrador (mostrar claramente qué campos y qué agregaciones se usaron en cada cálculo, especialmente cuando el resultado certificado proviene del mensual aprobado).
